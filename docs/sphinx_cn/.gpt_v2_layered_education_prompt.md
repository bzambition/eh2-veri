### 5.2 §4「源码逐段精读」的写法

错误示范：
> `eh2_ifu_compress_ctl` 模块实现压缩指令展开。它接收 16-bit C 扩展指令，
> 输出对应的 32-bit 标准 RV32I 指令。

正确示范：
````rst
**`eh2_ifu_compress_ctl.sv` 的核心：``c2i`` 展开函数**

.. code-block:: systemverilog
   :caption: design/ifu/eh2_ifu_compress_ctl.sv:L42-L68
   :linenos:
   :emphasize-lines: 5, 12

   always_comb begin
     o_legal = '1;
     o_instr32 = '0;
     case (i_instr16[1:0])
       2'b00: begin  // C0 quadrant: c.addi4spn / c.lw / c.sw ...
         case (i_instr16[15:13])
           3'b000: begin  // c.addi4spn
             if (i_instr16[12:5] == 8'b0) o_legal = '0;
             o_instr32 = {2'b00, i_instr16[10:7], i_instr16[12:11],
                          i_instr16[5], i_instr16[6], 2'b00,
                          5'h02, 3'b000, 2'b01, i_instr16[4:2], 7'h13};
           end
           ...

**逐行讲解**：

* **L1** ``always_comb`` —— 这是**组合逻辑** ，不带时钟。压缩展开必须当周期完成，
  下一级流水（F2 译码）马上要用，挂在寄存器后会延迟一周期。
* **L2-3** 默认值赋 `1` / `0` —— SV 编码风格：所有 ``output`` 在 ``always_comb`` 入口
  先赋默认值，避免 ``case`` 不全覆盖时综合出锁存器（latch），这是 ``set_assign_scoring``
  期望的写法。
* **L4** ``i_instr16[1:0]`` —— RV C 扩展规范规定低 2 位编码 quadrant（00/01/10 是 C 指令，
  11 是 32-bit 标准指令）。看 ``riscv-spec.pdf`` 第 16 章表 16.6。
* **L5 (高亮)** ``2'b00`` —— C0 quadrant，包含 ``c.addi4spn`` / ``c.lw`` / ``c.sw`` 等
  以栈指针为基址的内存指令。
* **L8** ``8'b0`` 检测 —— C 规范规定 ``c.addi4spn imm=0`` 是非法的（reserved），
  这里显式判出来送 ``o_legal=0``。
* **L9-11 (高亮)** —— 这是**位拼接**展开。EH2 把 16-bit 编码的 8-bit 立即数
  按 RV-C 规范的非线性映射重排到 32-bit ``addi`` 的 ``imm[11:0]`` 字段。
  对照 ``riscv-spec.pdf`` 表 16.5 看映射关系（``nzuimm[5:4|9:6|2|3]``）。

**为什么这么写而不是用查找表（ROM）？**

候选方案 A：用 ROM（256 项 × 32-bit）—— ASIC 综合后约 1100 cells，且关键路径
经过 ROM 译码器（约 0.8ns @ 28nm）。

候选方案 B（当前实现）：直接位拼接 + ``case`` —— ASIC 综合后约 350 cells，
关键路径 0.4ns。Espresso 工具能进一步最小化 ``case`` 表达式。

EH2 选 B，原因：取指路径是 ASIC 时序最紧的路径之一（与 BHT 预测并行），
省 0.4ns 时序余量很关键。Ibex 走的是同样路径，参见
``/home/host/ibex/rtl/ibex_compressed_decoder.sv:L<行号>``。
````

## 八、写作规范（与 v1 一致，重要规则重申）

### 8.1 RST 规范
- 一级 ``====``，二级 ``----``，三级 ``~~~~``，四级 ``^^^^``
- 章节元数据头三行：``:status:`` ``:last-reviewed:`` ``:authors:``
- 中文：技术名词首次出现给英文括注（如"验证环境（verification environment, env）"）
- 工业术语保留英文：UVM、VCS、URG、IMC、Formality、ncelab、irun、DC、LEC、SVA、cosim、ISS、RVFI、DPI、riscv-dv、cover.cfg、ucdb、vdb、INCA_libs、simv 等
- 标点中英混排时优先中文标点，代码内保留原样

### 8.2 代码引用规范
- 优先 ``.. literalinclude:: <绝对路径>`` 而非粘贴（保证与代码同步）
- 路径用项目根的绝对路径 ``/home/host/eh2-veri/...``
- 行号范围用 ``:lines: 40-68`` 而非 ``:lineno-match:``（更稳定）
- ``:emphasize-lines:`` 高亮关键行
- ``:caption:`` 给"file_path:L<起>-L<止>"格式

### 8.3 数据与命令规范
- 所有数字必须可追溯（line 95.05% 来自 ``build/signoff_vcs/cov_merged/dashboard.txt``）
- 所有命令必须可在本机跑通；先在本机 dry-run 验证
- 所有路径必须在仓库中真实存在（``ls`` 检查）
- 涉及未来工作 / 未实现的，用 ``.. todo::`` 块隔离，不混进正文

### 8.4 中文表达规范
- 不写"显然"、"易知"、"读者应该明白"
- 主动语态优先："IFU 取指" > "指令被 IFU 取得"
- 不滥用术语缩写，每个缩写首次出现给完整中英对照
- 段落长度：技术段落 3-5 句一段，叙事段落可长，列表项 1-3 行

§1 为什么是 EH2 而不是其它 RISC-V 核
-----------------------------------

如果你只想验证一个 RV32IMC 单核处理器，业内有数十款开源选择：Ibex、CVA6、
Hummingbird E2、SweRV EH1、Picorv32……为什么 EH2？

EH2 的独特之处在于 **双硬件线程（dual-threaded）** ：在一个流水线里轮转两个
独立的执行上下文（thread 0 / thread 1），共享 ICache/DCCM 但拥有独立的 PC、
寄存器堆、CSR、PMP 配置。这种"双 thread 单流水线"设计在工业里少见但有真实
用武之地：

* 工业控制器：thread 0 跑实时控制环路，thread 1 跑诊断与日志，两者不互相阻塞
* 网络处理：thread 0 跑数据面，thread 1 跑控制面

EH2 把这种设计开源，给学界一个**研究多线程验证方法学**的真实工业级目标。
EH2 验证比单线程难在哪里？详见 :doc:`/02_core_reference/dual_thread` § 1。

§4 源码逐段精读：从 ``eh2_veer_wrapper.sv`` 入手
------------------------------------------------

EH2 验证平台的第一个"读者要看的文件"是顶层 wrapper。它告诉你**核要连接到哪些
信号** ，验证平台就需要驱动/采样这些信号。

.. code-block:: systemverilog
   :caption: /home/host/Cores-VeeR-EH2/design/eh2_veer_wrapper.sv:L40-L80
   :linenos:
   :emphasize-lines: 5, 12, 20

   module eh2_veer_wrapper
   import eh2_pkg::*;
   #(
     `include "eh2_param.vh"
   )(
     input  logic                     clk,
     input  logic                     rst_l,
     input  logic [pt.PIC_TOTAL_INT_PLUS1-1:0] extintsrc_req,
     ...
     // ICCM SRAM 接口 (32-bit data, 1-cycle latency)
     input  logic [pt.ICCM_NUM_BANKS-1:0]      iccm_rd_data,
     output logic [pt.ICCM_NUM_BANKS-1:0]      iccm_rd_en,
     ...
     // AXI4 master 接口
     output logic                              lsu_axi_awvalid,
     input  logic                              lsu_axi_awready,
     ...

**逐行讲解**：

* **L1-2** ``module ... import eh2_pkg::*`` —— SystemVerilog 推荐写法：模块定义里
  ``import`` package，让模块内所有 typedef 可直接用，不用每个变量都加 ``eh2_pkg::``
  前缀。``eh2_pkg`` 定义在 ``design/include/eh2_pkg.sv``，存放 ``trap_t`` /
  ``alu_pkt_t`` 等 typedef。
* **L3-5 (高亮)** ``#( \`include "eh2_param.vh" )`` —— 把所有参数（ICache 大小 / DCCM 大小 /
  线程数等）通过 ``.vh`` 头文件方式注入到 parameter port。参数表里有 50+ 项，
  在 ``design/include/eh2_param.vh`` 集中维护。验证平台在 ``core_eh2_tb_top.sv``
  实例化时覆盖这些参数来配置 DUT 形态。
* **L8** ``extintsrc_req`` —— 外部中断输入，宽度由 ``PIC_TOTAL_INT`` 参数决定。
  默认 31 个中断源。这是 IRQ agent 的驱动目标，见 :doc:`/05_verification_arch/agent_irq`。
* **L12 (高亮)** ICCM 接口 —— 这是"指令紧耦合内存"，0 cycle access。这是 EH2 vs Ibex
  的关键差异（Ibex 无 ICCM）。TB 在 ``core_eh2_tb_top.sv`` 用一个简单的 ``$readmemh``
  + 寄存器数组来模拟 ICCM SRAM。
* **L20 (高亮)** AXI4 master 接口 —— 这是 LSU 对外的 AXI4 总线，是 AXI4 agent 的
  监控/响应目标。AXI4 信号定义参见 :doc:`/02_core_reference/bus_axi_ahb`。

**为什么 EH2 用 ``logic`` 而不是 ``reg``/``wire``？**

候选方案 A：``reg`` / ``wire`` 显式区分 —— 传统 Verilog 风格，需要在
``always`` 块和 ``assign`` 之间手动选对，编译期才报错。

候选方案 B（当前）：``logic`` 统一 —— SV 引入的"通用单 bit 类型"，编译器自动推断
是 wire 还是 reg。代码更简洁，但要求工具支持 SV-2005 +。

EH2 全代码用 ``logic``，这要求 lint（Verible）按 SV-LRM 1800-2017 规则检查，
参见 ``lint/verible.rules``。
